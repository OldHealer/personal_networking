## Обзор AI‑агентов в проекте Rockfile

Этот файл описывает целевые AI‑агенты и системы, которые разработаны/будут разрабатываться поверх Rockfile с использованием **LangGraph** и **MCP**.  
Фокус: **агентные пайплайны и оркестрация инструментов** (пока без RAG).

---

## Агент 1. Multi‑step Task Agent: "Подготовь к встрече с контактом" (реализован)

**Цель:** показать классический многошаговый пайплайн LangGraph: стабильный control flow + LLM‑узлы в реальном приложении Rockfile.  
Пример задачи:
> «Подготовь меня к встрече с Иваном»

### 1.1. Поведение агента

Шаги:

1. **Выбрать контакт**
   - в **текущем** реализации пользователь уже выбрал человека в UI (автодополнение по списку из `GET /api/v1/contacts`); в агент уходит готовый **`contact_id`**;
2. **Получить историю** (`get_history` → `contacts_get` / `contacts_get_tool`)
   - загрузить **карточку** (`contact`), **связи** (`links`) и **взаимодействия** (`interactions`);
   - контекст «что было в последней встрече» и краткие формулировки сути общения;
   - связи между людьми и контекст знакомства — из **`links`** (`relationship_type`, `context`) и осмысленных полей карточки (`relationship_type`, блок личных данных, цели).
3. **Выделить важное**
   - кратко суммировать:
     - кто это (роль, контекст знакомства);
     - о чем говорили в прошлые разы;
     - какие были обещания/ожидания с обеих сторон;
     - общие знакомые и «мостики».
4. **Сгенерировать советы**
   - предложить темы для разговора;
   - напомнить про обещания и возможные follow‑up;
   - при необходимости — накидать черновик сообщения/письма контакту.

### 1.2. Простой LangGraph‑pipeline

Логика без ветвлений (ноды):

- **START**
- `get_history`  
  Действия:
  - вызывает MCP `contacts.get` (карточка + связи + взаимодействия по контакту);
  - укладывает всё в `state.history` в нормализованном виде:
    - `contact` (основные поля);
    - `links` (связи с другими людьми);
    - `interactions` (последние N взаимодействий).
- `summarize_history` (LLM‑узел)  
  Действия:
  - делает короткое резюме «кто это» и «что было ранее»;
  - выделяет обещания, конфликты, чувствительные темы;
  - формирует список удобный для показа в UI.
- `generate_advice` (LLM‑узел)  
  Действия:
  - на основе `summary`, исходного `user_query` и истории формирует:
    - список советов по встрече;
    - возможные вопросы к человеку;
    - напоминания о follow‑up после встречи;
    - опционально — черновик письма/сообщения.
- `format_output`  
  Приводит ответ к удобному формату для фронта/CLI (markdown‑подобная структура с блоками: «Кратко», «Кто это», «О чём говорить», «Не забыть», «Черновик сообщения»).
- **END**

### 1.3. Структура состояния (state) агента

Минимальный вариант `state`:

- `user_query: str` — исходный запрос пользователя;
- `contact_id: str` — выбранный контакт (UI выбирает из списка, бэкенд не ищет по имени);
- `history: { contact, links, interactions }` — доменные данные из Rockfile;
- `summary: { profile, last_interactions, promises, risks }` — результат `summarize_history`;
- `advice: { talking_points, followups, draft_message }` — результат `generate_advice`;
- `output: str` — финальный отформатированный ответ для UI.


### 1.4. Переключение режима (MCP vs Local)

Агент поддерживает два режима получения данных (узел `get_history`):

- **`ROCKFILE_AGENT_MODE=mcp`** (по умолчанию)  
  LangGraph‑узел вызывает MCP tool `contacts_get_tool`, который валидирует JWT и читает данные из БД.

- **`ROCKFILE_AGENT_MODE=local`**  
  LangGraph‑узел вызывает `contacts_tools.contacts_get(...)` напрямую через сервисный слой (без MCP).

Это было полезно для обучения:
- MCP режим показывает “каноничную” tool‑архитектуру
- local режим проще для отладки и быстрее

## Агент 2. «Concierge» личной сети 

**Статус:** в этом репозитории описан **продуктово‑технический дизайн** и контракт графа.

**Зачем отдельный агент.**
Агент 1 показывает **линейный** pipeline (данные → суммаризация → совет по одному контексту). Агент 2 показывает **оркестрацию**: классификация намерения, **ветвление**, **циклы** «мало данных → ещё одна загрузка через tool», явная **аудитория tool‑вызовов** для демонстрации. 

### 2.1. Три пользовательские функции (один агент — разные ветки графа)

После ввода сообщения первый «умный» узел определяет **intent** и переводит выполнение в нужную ветку (п. 2.3).

#### A. Напоминания о днях рождения

- **Запросы:** «кому скоро ДР?», «на этой неделе дни рождения» и т.п.
- **Данные:** `ContactCard.birthday`; при развитии продукта - ещё `ContactFamilyMember.birthday` (отдельная сущность в модели уже есть).
- **Логика:** узел **детерминированно** строит список контактов в окне дат. LLM при желании **формулирует** человекочитаемый текст, **не добавляя** фактов, которых нет в данных.

#### B. Сводка по обещаниям: «я должен» и «мне должны»

- **Запросы:** «что я кому обещал», «кто мне что должен», «открой мои открытые обещания».
- **Источники в Rockfile сегодня:**
  - агрегат **`ContactCard.promises`** (JSON: текст обещания, `completed_at`, связь с `interaction_id` при наличии);
  - детализация в **`ContactInteraction.promises`** и текстах **`notes`** по разговорам.
- **Честное ограничение модели:** отдельного поля «обязательство исходит от меня / от контакта» в БД **пока нет**. Целевой UX «две колонки» достигается так:
  - на первом этапе — **LLM классифицирует** каждую открытую строку по формулировке (с жёсткими guardrails: «если нельзя однозначно — пометь как *неясно*»);
  - на следующем этапе разработки — явная схема, например `direction` / `owed_by` в элементе promise (см. §2.7).
- **Поведение графа:** узел **собирает** все незакрытые обещания по нужным контактам → LLM **группирует** в отчёт с двумя разделами + секция «требует уточнения».

#### C. Поиск контакта под цель

- **Пример:** «Найди контакт, кто может проконсультировать по юридическому вопросу».
- **Поведение:**
  1. Узел (LLM или правила) выделяет **критерии** из текста: домены, роли, ключевые слова.
  2. **`contacts.list`** — получить пул кандидатов (на REST уже есть `GET /api/v1/contacts`, в т.ч. с **`last_interaction_at`**).
  3. Для топ‑K кандидатов — **`contacts.get`** (карточка + `interests` + заметки в `hobbies` + `links` + последние `interactions`), чтобы не гонять тяжёлый контекст по всей базе.
  4. LLM **ранжирует** и **объясняет** топ‑3; ссылки только на реальные поля JSON.
  5. Если данных мало — срабатывает **цикл** enrich (см. §2.3): другой фильтр списка, догрузка **связей** (`ContactLink`) для «второго кольца» знакомых и т.д., до лимита итераций.

### 2.2. Почему граф намеренно **не линейный**

| Механизм | Что демонстрирует в портфолио |
|----------|-------------------------------|
| **`route_intent`** после ввода пользователя | Маршрутизация: ДР / обещания / поиск / уточнение |
| **Условные рёбра** | «низкая уверенность в intent → спросить уточнение» |
| **Цикл `gather ↔ judge`** | Пока `context_sufficient == false` и `tool_round < max` — ещё вызовы tools |


Таким образом.

Агент 1 = «цепочка». Агент 2 = «**мини‑операционка**» над данными с явным лимитом цикла (`max_tool_rounds`), чтобы граф всегда останавливался.

### 2.3. Черновик топологии графа (контракт для реализации)

Текстовая схема — ориентир для LangGraph `StateGraph`, не код.

```text
START
  → ingest_user_input          # нормализация текста, извлечение периода для ДР и т.д.
  → route_intent               # birthdays | promises | matchmaker | unknown

route_intent
  → [birthdays]   collect_birthdays_window
  → [promises]    collect_open_promises
  → [matchmaker]  extract_criteria → shortlist_from_list → enrich_context (цикл)
  → [unknown]     ask_clarification  → END или возврат к route_intent

enrich_context (цикл)
  → need_more = f(state)       # эвристика: мало полей / мало кандидатов
  → IF need_more AND tool_round < max THEN contacts.get / links … ; tool_round++
  → ELSE rank_and_explain     # LLM только по переданным фактам

  → synthesize_report          # единый markdown / JSON для UI
  → (опционально) hitl_confirm_writes
END
```

### 2.4. Черновик полей `state` (расширяемый TypedDict)

| Поле | Назначение |
|------|------------|
| `user_message: str` | Сырой ввод из чата / формы |
| `intent` | Результат маршрутизации: `birthdays` / `promises` / `matchmaker` / `unknown` |
| `intent_confidence: float` | Для ветвления «уточнить вопрос» |
| `time_window` | Для сценария ДР (например N дней вперёд) |
| `criteria: dict` | Извлечённые критерии для matchmaker |
| `candidate_ids: list` | Короткий список после отбора |
| `snapshots: dict` | Кэш ответов tools, чтобы не дублировать вызовы |
| `tool_round: int` | Счётчик итераций цикла |
| `audit: list` | Краткий журнал вызовов tools |
| `final_reply: str` | Ответ пользователю |

### 2.5. Инструменты (MCP целевые ↔ REST сегодня)

| Tool (цель) | Роль в Агенте 2 | Что есть в проекте сейчас |
|-------------|-----------------|---------------------------|
| `contacts.list` | Пул и фильтрация кандидатов | `GET /api/v1/contacts` (+ `last_interaction_at`) |
| `contacts.get` | Полный контекст по контакту | Логика как в `contacts_get` / Агент 1 |
| `interactions.list` | История формулировок и обещаний | Эндпоинты по контакту уже есть |
| `links.list` | «Второе кольцо», связи между людьми | Эндпоинты links |
| `promises.open` *(желательно)* | Один вызов «все открытые обещания» | *Пока нет* — сбор из списка контактов + детализации |
| `family.birthdays` *(желательно)* | ДР семьи одним запросом | *Пока нет* — можно композировать из `contacts.get` после расширения API |

**Режим первой реализации:** только **чтение** и отчёты (без автозаписи в БД). 

### 2.6. Резюме 

> Агент 2 — это **мультисценарный "concierge"** над теми же данными Rockfile: дни рождения, сводка обязательств, интеллектуальный shortlist людей под задачу пользователя. В отличие от Агента 1, граф **ветвится по намерению** и может **в цикле** догружать контекст через tools до порога или лимита; LLM не заменяет БД, а **интерпретирует** уже выбранные факты.

### 2.7. Дальнейшее развитие модели обещаний (когда понадобится)

- Явное поле направления обязательства (`owed_by: self|contact|unknown`) в JSON `promises`.
- Отдельный агрегирующий API / MCP `promises.open` для одного roundtrip.
- **Router верхнего уровня** («один чат Rockfile»): делегировать в Агент 1 vs Агент 2 — либо отдельный граф, либо объединение с `route_intent`.
- **RAG** по текстам взаимодействий — после стабилизации tools.
